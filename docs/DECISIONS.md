## Phase 2 — Day 1: repo, environment, seeded schema

### Tooling: `uv` for env + dependencies
- **What:** `uv` manages the venv, `pyproject.toml`, and `uv.lock`.
- **Why:** fast, reproducible lockfile, modern standard; reads well in a portfolio repo.
- **How:** `uv sync` builds the venv; `uv run python <script>` executes. `package = false`
  in `[tool.uv]` — this is an application, not a distributable library, so nothing to build yet.
- **Gotcha:** on this Windows box `python`/`pip` resolve to the Microsoft Store stub
  (App Execution Alias) and fail. The real interpreter is the `py` launcher
  (`C:\Users\piero\AppData\Local\Programs\Python\Python311`). `uv` was installed with
  `py -m pip install --user uv` and lives in `...\Roaming\Python\Python311\Scripts`.

### Database: Postgres on Supabase
- **What:** Postgres via Supabase, not SQLite.
- **Why:** reuse the deploy pattern + the session-pooler gotcha as an interview story
  (carried over from project #1). Spec §3 recommended Postgres for exactly this.
- **Gotcha (carried, must hold):** `DATABASE_URL` must use the **Session Pooler (port 5432, IPv4)**.
  The Direct connection is IPv6-only and fails from Railway/Docker; the Transaction
  pooler (6543) breaks prepared statements.

### Schema: 4 fixed tables, identity PKs
- **What:** `customers`, `products`, `orders`, `order_items` per spec §6. No extra tables in v1.
- **How:** `BIGINT GENERATED BY DEFAULT AS IDENTITY` (not `ALWAYS`) so the seed can insert
  explicit ids and wire foreign keys deterministically; the seed re-syncs the identity
  sequences with `setval` afterward. `status` constrained by CHECK to a fixed enum set.
  Indexes on the FK + date columns the agent will filter/join on.
- **Note:** `sessions` (multi-turn memory) is **v2**, deliberately not created now.

### Seed: deterministic, medium volume, anchored to "today"
- **What:** ~200 customers, ~50 products, ~5,000 orders over 6 months, ~15k order_items.
- **Why "medium":** enough for top-N, aggregations, and "inactive 90 days" without bloating the DB.
- **How:** `random` + `Faker` seeded with `SEED=42` → reproducible shape. Order timestamps are
  anchored to the real `now()` at run time so relative questions ("last month", "last 90 days")
  stay meaningful even though the seed is deterministic. `seed.py` TRUNCATEs first → idempotent.

### Read-only role created on Day 1
- **What:** `analyst_ro` DB role with LOGIN + SELECT-only on `public`.
- **Why:** "read-only in depth" — the defense lives at the connection layer, not just the prompt
  (spec §3). Created now so the runtime URL is ready from the start.
- **How:** `db/create_readonly_role.py` grants USAGE + SELECT (+ default privileges for future
  tables), explicitly REVOKEs INSERT/UPDATE/DELETE/TRUNCATE, and prints `DATABASE_URL_RO`.
- **Gotcha (confirmed working):** Supabase's session pooler expects a custom role's username as
  `<role>.<project-ref>` (e.g. `analyst_ro.vmzcojbvsjhsdsqakpsy`), **not** `analyst_ro` alone —
  auth fails otherwise. `create_readonly_role.py` derives the RO URL by swapping
  `postgres` → `analyst_ro` in the admin username, preserving the `.<ref>` suffix.
- **Verified:** `db/check_readonly.py` connects as `analyst_ro`, SELECT returns 200 customers,
  INSERT is rejected with `permission denied for table customers`. Read-only holds at the
  connection layer, not just in the prompt.

### Region check (us-west-1)
- **What:** Supabase project host is `aws-1-us-west-1.pooler.supabase.com` → region us-west-1.
- **Why it matters:** São Paulo (sa-east) reintroduces ~3.5s cross-region overhead (project-#1 pain).
  us-west-1 confirmed before seeding. The `aws-1` prefix is just the pooler instance number, not the region.

### Minor: `table` is a reserved word
- `verify.py` aliased a count column `AS table`, which Postgres rejects in `ORDER BY`. Renamed to
  `table_name`. Noted so future generated SQL avoids reserved identifiers unquoted.

### External lead-time accounts activated Day 1
- **What:** Supabase project + Railway account created/activated today, even though Railway is
  used later. Mirrors the project-#1 lesson: start external lead-time on Day 1.
